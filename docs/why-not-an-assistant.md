# Why a Chief of Staff is Not an Assistant

If you set up this kit and then start using it like ChatGPT, you'll get back something that feels like ChatGPT. The framing is everything.

## The difference, sharp

**An administrative assistant** waits to be told what to do. The default question is "what task should I do for you?" The relationship is transactional. You hand it work, it returns work.

**A second brain** stores what you put in. The default question is "what would you like to capture?" The relationship is custodial. You give it information, it gives it back when you ask.

**A notetaker** records what happened. The default question is "what should I write down?" The relationship is observational. You talk, it captures.

**A Chief of Staff** holds the picture. The default question is "what's the picture, what's drifting, what's next?" The relationship is operational. You're running something together, and they help you run it well.

## What that looks like in practice

A founder walks into a Monday session with their assistant: *"Pull up the proposal for Acme."*

A founder walks into a Monday session with their chief of staff: *"Acme is closing on Wednesday — you owed them revised scope and pricing by EOD Friday and I haven't seen it land. Want to review the draft now or push to Tuesday morning? Also, your Q3 commitments call with the board is at 4pm, and you haven't touched the deck since Thursday. The Nordis check we were expecting Friday hasn't hit yet — Engagement PM is on it. Health side: you're at three workouts this week against five, with two days left."*

Same founder, same hour, same toolkit. The first one returns the proposal. The second one returns the proposal, plus the dropped Friday commitment, the deck that hasn't been touched, the slipping cash collection, and the off-pace fitness goal.

That's the gap this kit is trying to close.

## What it requires of you

The CoS relationship works when you treat it like one:

- **Tell it what's going on.** State files don't update themselves. After a conversation, drop the recap. After a decision, log it. The CoS is reading these to decide what to surface next.
- **Let it surface, don't just answer.** When it flags drift, take it seriously even if it's slightly off — the alternative is silence.
- **Push back on tone drift.** If the CoS slides into "what would you like me to do for you?", say so. Say "you're being an assistant again. What's drifting?" It will recalibrate.
- **Treat personal and health with the same weight.** This is the rule that does the most work. Most people set up an operating system that ignores everything except client work. The CoS only delivers on the "Chief of Staff" framing if family commitments, sleep, training, faith, and family situations get the same row in state/current.md as the pipeline.

## What it stops doing once you set it up right

- Asking what's on your calendar (it reads it).
- Asking what your priorities are (they're in state).
- Asking who the players are (they're in `people/`).
- Asking how you want to be talked to (it's in `.claude/rules/communication-style.md`).
- Asking what was open from last time (it reads `state/last-session.md`).

A good session start sounds like: *"You said yes to the Acme scope last night — want me to log the decision and draft the email? Also flagging that the Nordis Friday check still hasn't hit. Heath is OK pace — three down, two to go."* Not: *"What would you like to work on today?"*

## The through-line

Every file in this kit, every section in the setup interview, every rule in `.claude/rules/` is pointing at the same thing: the agent is a chief of staff, not an assistant.

If at any point during setup or operation the system drifts toward "what task can I do for you" — push back. That's an administrative assistant, and there are plenty of those. The Chief of Staff is the one that holds the picture and tells you what's drifting before you ask.
